# /summary — Session Summary

Synthesize the current session into a markdown file and a self-contained interactive HTML file, then open the HTML in Safari via lavish-axi for live annotation.

## Step 0 — Resolve the label

Read `$ARGUMENTS`. If a label was supplied, use it directly. Otherwise:

```bash
git branch --show-current 2>/dev/null
```

Strip common prefixes (`feat/`, `fix/`, `chore/`, `docs/`), replace `/` and `_` with `-`, lowercase the result. If the branch name is empty or the command fails, use `session`.

Capture today's date:

```bash
date +%Y-%m-%d
```

Construct the output paths:
- `docs/summaries/<label>-<YYYY-MM-DD>.md`
- `docs/summaries/<label>-<YYYY-MM-DD>.html`

Ensure the directory exists:

```bash
mkdir -p docs/summaries
```

## Step 1 — Synthesize the session

Review the full conversation context and produce content for each of these sections. Omit a section if it has no content (Synod Council Findings if no agents ran; SDD Task Status if no SDD work occurred).

**Overview** — 1-2 sentences describing what the session accomplished, including branch or project context.

**Decisions Made** — bulleted list of concrete decisions with a brief rationale for each.

**Synod Council Findings** — if any Synod Council agents were dispatched, list each agent's name and a brief summary of their findings. Omit this section entirely if no agents ran.

**SDD Task Status** — if any SDD tasks were active during the session, list each task with its status: `pending`, `in-progress`, or `complete`. Omit this section entirely if no SDD work occurred.

**Open Questions** — unresolved threads, known unknowns, and recommended next steps.

## Step 2 — Write the markdown file

Write `docs/summaries/<label>-<date>.md` with standard markdown headers (`##`) for each section. Include the session label and date in the document title (`# <label> — <date>`).

## Step 3 — Write the HTML file

Write `docs/summaries/<label>-<date>.html` as a self-contained, interactive HTML document.

Requirements:
- Include `<meta name="lavish-live-reload" content="root">` in `<head>` so lavish-axi watches the full file for changes.
- Load Tailwind CSS from CDN: `https://cdn.tailwindcss.com`
- Load DaisyUI from CDN: `https://cdn.jsdelivr.net/npm/daisyui@latest/dist/full.min.css`
- Structure:
  - Page header: session label + date, styled as a hero or card header
  - One `<details open>` block per content section, with `<summary>` as the section title
  - For **SDD Task Status**: render each task as a DaisyUI badge with cycling status. Badge colors: `badge-neutral` for pending, `badge-info` for in-progress, `badge-success` for complete. Include an inline `onclick` handler on each badge that cycles through the three states (pending -> in-progress -> complete -> pending) and updates both the badge text and its CSS class accordingly.
- All JavaScript must be inline `<script>` tags — no external scripts.
- Include a `<footer>` at the bottom: `Generated by /summary · Open with: lavish-axi <this-file>`

Example badge cycling script (inline, adapt as needed):

```html
<span
  class="badge badge-neutral cursor-pointer"
  data-status="pending"
  onclick="
    const states = ['pending','in-progress','complete'];
    const colors = ['badge-neutral','badge-info','badge-success'];
    const i = states.indexOf(this.dataset.status);
    const next = (i + 1) % 3;
    this.dataset.status = states[next];
    this.className = 'badge ' + colors[next] + ' cursor-pointer';
    this.textContent = states[next];
  "
>pending</span>
```

## Step 4 — Open in Safari via lavish-axi

Run this shell sequence exactly:

```bash
html_file="docs/summaries/<label>-<date>.html"
session_url=$(lavish-axi "$html_file" --no-open 2>&1 | grep -o 'http://[^[:space:]]*' | head -1)
if [ -n "$session_url" ]; then
  open -a Safari "$session_url"
else
  open -a Safari "$html_file"
fi
```

Capture `session_url` for the report in Step 5. If `lavish-axi` is not found or errors, fall back silently to `open -a Safari "$html_file"` and note the fallback in the report.

## Step 5 — Report to the user

Confirm:
- Path to the markdown file written
- Path to the HTML file written
- Safari URL if captured (or fallback used)

Then print this message verbatim (substituting the actual html_file path):

> "To iterate interactively: I can call `lavish-axi poll <html-file>` to receive your annotations from Safari, update the HTML, and the page will auto-reload. Say 'poll lavish' when you want me to listen for your feedback. Say 'end lavish' when done."
